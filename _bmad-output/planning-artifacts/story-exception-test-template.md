# Story Exception Test Template (Story 2.1 / 4.1 / 4.2 / 6.2 / 7.2)

## 목적

- Correct Course에서 보강한 예외/오류 AC를 구현 전에 검증 가능한 테스트 케이스로 고정한다.
- 본 문서는 템플릿이며, 구현 상세 API/엔드포인트 확정 후 입력값만 치환해 사용한다.

## 공통 규칙

- 케이스 ID 형식: `STORY-ID-ERR-XX`
- 최소 포함 필드: 사전조건, 입력/행동, 기대결과, 관측 포인트(로그/이벤트/응답)
- 오류 응답 검증: `code`, `message`, `recovery_guide` 포함 여부 확인
- 이벤트/로그 검증: 요구된 경고/오류 이벤트가 정확히 1회 기록되는지 확인

## Story 2.1 테스트 템플릿

### S2.1-ERR-01 입력 유효성 위반 (400)
- 사전조건: 판단 카드 생성 API 접근 가능
- 입력/행동: 필수 필드 누락 또는 타입 불일치 요청 전송
- 기대결과:
  - HTTP 400
  - 응답에 `code`, `message`, `recovery_guide` 포함
  - 비정상 데이터가 저장되지 않음
- 관측 포인트: API 응답 스키마, DB write 없음

### S2.1-ERR-02 데이터 신선도 임계치 초과
- 사전조건: 의향 데이터 `stale` 상태로 설정
- 입력/행동: 판단 카드 생성 API 호출
- 기대결과:
  - fallback 판단 결과 반환
  - `decision_card_error` 이벤트 기록
  - 사용자 플로우는 차단되지 않음
- 관측 포인트: 이벤트 로그, 응답 payload의 fallback indicator

## Story 4.1 테스트 템플릿

### S4.1-ERR-01 외부 신호 지연/타임아웃
- 사전조건: 외부 신호 소스 응답 지연 시뮬레이션
- 입력/행동: 신호 수집 배치/스트림 실행
- 기대결과:
  - 마지막 검증 신호로 degraded 처리
  - 지연 메트릭 기록
  - 판단 플로우 정상 유지
- 관측 포인트: 지연 메트릭, degraded flag, 사용자 여정 성공률

### S4.1-ERR-02 스키마 불일치 신호 유입
- 사전조건: 계약과 다른 필드를 가진 이벤트 준비
- 입력/행동: 수집 파이프라인 주입
- 기대결과:
  - 불일치 신호는 격리 큐로 이동
  - 핵심 처리 큐 오염 없음
  - 판단 API 정상 응답 유지
- 관측 포인트: 격리 큐 건수, 메인 큐 에러율, API 상태

## Story 4.2 테스트 템플릿

### S4.2-ERR-01 무권한 규칙 변경 (403)
- 사전조건: 권한 없는 사용자 토큰 준비
- 입력/행동: 알림 규칙 수정 요청
- 기대결과:
  - HTTP 403
  - 규칙 변경 미적용
  - 감사 로그 기록
- 관측 포인트: 알림 규칙 저장소 변경 없음, 감사 로그 entry

### S4.2-ERR-02 규칙 저장 실패 롤백
- 사전조건: 저장소 오류(일시 장애) 시뮬레이션
- 입력/행동: 알림 규칙 수정 요청
- 기대결과:
  - 기존 규칙 유지
  - 사용자에게 복구 가이드 노출
  - 재시도 가능 상태
- 관측 포인트: 이전 규칙 동일성, UI 오류 안내, 재시도 동작

## Story 6.2 테스트 템플릿

### S6.2-ERR-01 할당 키 누락/충돌
- 사전조건: user assignment key 누락 또는 충돌 데이터 준비
- 입력/행동: 실험 할당 처리 실행
- 기대결과:
  - 재할당하지 않고 요청 격리
  - `allocation_mismatch` 경고 기록
  - 기존 사용자 할당 불변
- 관측 포인트: 경고 로그, 격리 큐, 기존 할당 테이블 diff 없음

### S6.2-ERR-02 NFR21 임계치 미달
- 사전조건: 할당 일치율을 임계치 미만으로 만드는 테스트 데이터
- 입력/행동: 실험 리포트 집계 실행
- 기대결과:
  - 실험 리포트 집계 자동 중지
  - 운영 경고 알림 발생
  - 재개 조건이 명시됨
- 관측 포인트: 집계 job status, 경고 채널 알림, 운영 로그

## Story 7.2 테스트 템플릿

### S7.2-ERR-01 계약 스키마 위반
- 사전조건: 파트너 요청/응답 스키마 불일치 케이스 준비
- 입력/행동: 신호 조회 API 호출
- 기대결과:
  - 버전드 오류 규약 `schema_version_mismatch` 반환
  - 오류 응답 포맷 일관성 유지
- 관측 포인트: API 응답 코드/본문, 계약검증 리포트

### S7.2-ERR-02 쿼터 초과/레이트 리밋
- 사전조건: 파트너 쿼터 소진 상태
- 입력/행동: 추가 API 호출
- 기대결과:
  - 표준 제한 응답 반환
  - `Retry-After` 포함
  - 복구/재시도 가이드 제공
- 관측 포인트: 헤더 값, 오류 본문, 파트너 콘솔 메시지

# Story 1.2: 목적 불명확 진입과 상태 진단 플로우

Status: done

## Story

As a 목적이 불명확한 사용자,  
I want 현재 상태 진단 후 다음 한 단계 행동을 즉시 제안받고,  
so that 막힘 없이 탐색을 시작할 수 있다.

## Acceptance Criteria

1. **Given** 사용자가 첫 진입 화면에 도달했을 때  
   **When** 목적/상품 명확도 질문에 응답하면  
   **Then** 시스템은 상태 진단 결과를 저장하고 다음 한 단계 행동을 1초 이내에 제시해야 한다.
2. **Given** 사용자가 모바일 화면(최소 390x844)에서 진입했을 때  
   **When** 진입 화면이 렌더링되면  
   **Then** `현재 상태 / 다음 행동 / 가역성` 정보가 initial viewport에 노출되어야 한다.
3. 상태 진단 저장 계약은 `intent_maturity`, `goal_confidence`를 포함하고 Story 2.x/3.x에서 그대로 재사용 가능한 DTO 형태여야 한다.
4. 상태 진단 저장 API는 `{ data, meta }` / `{ error: { code, message, details?, requestId } }` 규약을 지키고, `requestId`와 로그 `traceId`를 상관 가능해야 한다.
5. 상태 정보 전달은 색상 단독 금지 규칙을 지키고(텍스트/아이콘 병행), 키보드-only 탐색과 스크린리더 라벨이 동작해야 한다.
6. 화면 이벤트는 UI alias(`state_assessed`)를 사용하되, 저장/전송 시 PRD canonical(`view_landing`)으로 매핑되어 기록되어야 한다.

Coverage: FR1, FR2, FR5, FR6, UX-DR1, UX-DR2

### Epic Context

- Epic 1 목표: 목적이 불명확한 사용자도 탐색을 시작하게 만들고, 의향/목표 상태를 점진적으로 명확화.

### Previous Story Intelligence (Story 1.1)

- 이미 존재하는 경로: `/`, `/intent`, `/decision` 및 `feature/intent`, `feature/decision` 폴더 골격.
- 구조/규약은 이미 선언됨:
  - API 성공/오류 포맷: `{ data, meta }`, `{ error: { code, message, details?, requestId } }`

### Technical Requirements (Must Follow)

- 프레임워크/런타임: Next.js App Router + TypeScript 유지.
- 레이어 경계: `feature -> domain -> infra` 단방향, `feature -> infra` 직접 import 금지.
- 내부 통신 순서: `contracts -> adapters -> handlers -> UI`.
- API/이벤트 필드:
  - datetime은 ISO-8601 UTC 문자열.
  - `requestId`와 로그 `traceId` 상관관계 유지.
  - 이벤트는 UI alias와 canonical을 분리 관리하고, 저장 계층에는 canonical만 기록.
- 스타일 가드레일:
  - 인라인 스타일 확장 금지, 전역/모듈 스타일 또는 토큰 기반 스타일로 이관.
  - 상태 전달에 색상 단독 사용 금지.

# Story 1.2: 프리리스팅 상태 저장 및 수정

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매 사용자,  
I want 등록한 매물을 `판매중` 또는 `프리리스팅` 상태로 저장하고 이후 변경하고 싶다,  
so that 판매 결심 이전의 물건도 자연스럽게 관리할 수 있다.

## Acceptance Criteria

1. 생성 시 상태를 `판매중` 또는 `프리리스팅`으로 선택할 수 있다.
2. 저장 후 상태를 수정할 수 있다.
3. 수정 시각이 최신으로 갱신된다.

## Dev Notes

### Story Intent

- Story 1.2의 목표는 Story 1.1에서 만든 기본 등록 흐름 위에 `프리리스팅` 상태 개념을 자연스럽게 올리는 것이다.
- 사용자는 생성 시 상태를 고르고, 저장 후 다시 열어 상태를 변경할 수 있어야 한다.
- 이 스토리는 상태 선택/수정과 시각 갱신을 다루며, 필수 필드 오류 표현의 깊은 설계는 Story 1.3에서 다룬다.

### Scope Boundaries

- 포함:
  - 생성 시 상태 선택
  - 저장된 매물의 현재 상태 조회
  - 상세 화면에서 상태 수정
  - 상태 변경 시 `updatedAt` 갱신
- 제외:
  - 필수 필드 오류 카피의 본격 확장
  - `listing.created.v1` 관측 파이프라인 전체
  - AI 초안, 추천가, 자동 가격조정, 운영자 전용 UI

### Recommended Implementation Shape

- 권장 수정 범위:
  - `src/domain/prelisting-status/`
  - `src/domain/listing/`
  - `src/infra/listing/`
  - `src/feature/listing/components/listing-form.client.tsx`
  - `src/feature/listing/actions/create-listing.action.ts`
  - `src/app/listings/[listingId]/page.tsx`
  - 상태 수정용 action/component 파일
- 경계 규칙:
  - `feature -> domain -> infra`만 허용
  - Page/Component/Server Action에서 Prisma를 직접 호출하지 않는다
  - casing 변환이 필요하면 infra 경계에서만 처리한다

### UX and Accessibility Guardrails

- 상태 선택은 `판매중` / `프리리스팅` 두 옵션만 보여야 한다.
- 상태 선택은 설명이 길지 않은 익숙한 폼 패턴이어야 한다.
- 상태 정보는 색상 단독으로 전달하지 않는다.
- 저장 CTA와 수정 CTA는 명확한 위치를 유지하고, pending 동안 중복 제출을 막는다.

- 액션/저장소 테스트:
  - 생성 시 `initialStatus` 저장
  - 상세에서 상태 변경 저장
  - 저장소 실패 시 재시도 가능 상태 유지
- 브라우저 테스트:

  - `/listings/[listingId]`에서 상태 변경 후 최신 상태/수정 시각 반영

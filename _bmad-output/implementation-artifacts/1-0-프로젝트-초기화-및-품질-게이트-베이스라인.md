## Dev Notes

### Story Intent

- 이 스토리는 사용자 기능 구현이 아니라 Story 1.1~1.4를 안전하게 진행하기 위한 Enabler Story다.
- Story 1.0에서 구현해야 하는 것은 새 기능이 아니라 실행 환경, 품질 게이트, 최소 앱 셸이다.
- FR14~FR17과 관련된 AI/추천가/자동 가격조정/운영 전용 UI는 이 스토리 범위가 아니다.

### Current Workspace Reality

- 현재 저장소는 앱 셸이 없는 테스트 하네스 중심 상태다.
- [package.json](/C:/Users/ok.works/Projects/PreProduct/package.json)는 `preproduct-tests` 이름과 Playwright 중심 스크립트만 가지고 있다.
- [tsconfig.json](/C:/Users/ok.works/Projects/PreProduct/tsconfig.json)는 테스트 파일만 포함하고 있어 App Router 기준선으로 확장되어야 한다.
- [ci.yml](/C:/Users/ok.works/Projects/PreProduct/.github/workflows/ci.yml)는 typecheck + Playwright shard + burn-in 중심이며, Active MVP baseline인 `lint`, `unit`, `contract`, `perf-budget`는 아직 없다.
- `.nvmrc`는 `24.14.1`이고, 저장소 엔진 필드와 실제 앱/Prisma 호환 범위를 맞춰야 한다.
- `src/`, `prisma/`, `contract-check.yml`, `perf-budget.yml`는 현재 존재하지 않는다.

### Implementation Guardrails

- 아키텍처 기준선은 Next.js App Router + TypeScript + PostgreSQL + Prisma + pnpm이다.
- 코드 구조는 `feature -> domain -> infra` 순서만 허용한다.
- import는 상대 상위 경로 대신 `@/*` alias를 사용한다.
- 환경변수는 단일 진입점에서 Zod로 검증하는 구조를 향해 가야 하며, Story 1.0에서 최소 기반을 마련한다.
- MUI가 프로젝트 표준 UI 라이브러리이므로, Story 1.0에서 UI 셸을 만든다면 Tailwind가 아니라 MUI 기준으로 맞춘다.
- `"use client"` 경계는 interactive leaf에만 두고, App Router 기본은 Server Component로 유지한다.
- 기존 Playwright 자산과 helper 구조는 이후 앱 연결을 전제로 유지해야 하므로 삭제 또는 대체하지 않는다.

### Quality Gate Requirements

- `lint`: Next/TypeScript 규칙 + 프로젝트 컨텍스트의 금지 패턴을 차단해야 한다.
- `typecheck`: strict TypeScript, `noUncheckedIndexedAccess`, `noImplicitReturns`를 유지해야 한다.
- `unit`: Jest baseline을 사용하고, mock 규칙은 `jest.mock()` 최상단/`afterEach(() => jest.clearAllMocks())`를 따른다.
- `contract`: `listing.created.v1` 이벤트의 공통 필드(`eventId`, `occurredAt`, `traceId`, `schemaVersion`)와 payload 필드(`listingId`, `initialStatus`) 검증을 최소 기준으로 삼는다.
- `perf-budget`: 정량 임계값이 필요하다. Story 1.0에서는 최소 앱 셸에 대한 측정 가능 기준을 만들고, 이후 Story 1.1+에서 사용자 여정 기준으로 강화한다.
- 차단형 gate는 GitHub Actions 파일 존재만으로 충분하지 않다. 실패 시 PR merge가 막히는 이름과 실행 경로가 문서화되어야 한다.

### Recommended Implementation Approach

1. 루트 워크스페이스를 Next.js 앱 기준 구조로 맞춘다.
2. 테스트 하네스 자산을 앱 셸과 병합하되 Playwright 설정은 유지한다.
3. 최소 도메인 계약 자산(`listing.created.v1`)과 단위 테스트 baseline을 먼저 만든다.
4. CI를 `lint/typecheck/unit` 빠른 경로와 `contract/perf-budget` 독립 게이트로 분리한다.
5. 최종적으로 required checks 이름을 문서에 고정한다.

### Specific Risks To Avoid

- 새 `preproduct/` 폴더를 생성해 현재 저장소를 중첩 구조로 만드는 실수
- 현재 Playwright 하네스를 덮어써서 QA 자산을 잃는 실수
- `contract-check.yml`과 `perf-budget.yml`을 단순 성공 워크플로우로 추가해 거짓 신호를 만드는 실수
- `package.json` 엔진, `.nvmrc`, CI Node 버전을 서로 다르게 두는 실수
- baseline 스크립트 이름과 GitHub required check 이름이 달라 merge gate가 무력화되는 실수

### Testing Requirements

- 최소 검증 커맨드는 로컬과 CI에서 동일하게 실행 가능해야 한다.
- 단위 테스트는 정상/경계값/에러 케이스를 분리할 수 있는 구조여야 한다.
- 계약 검증은 이벤트 schema 또는 동일 수준의 계약 테스트로 구현한다.
- perf budget은 숫자가 명시된 실패 기준을 가져야 한다.
- 기존 `tests/e2e/framework-smoke.spec.ts`는 baseline smoke로 유지할 수 있지만, Story 1.0의 required checks 목록을 흐리면 안 된다.

### Project Structure Notes

- 아키텍처 문서의 초기화 명령은 새 디렉터리를 만드는 예시다. 현재 저장소는 이미 루트에 lockfile, tests, scripts가 있으므로 "동등한 구조를 현재 root에 적용"하는 것으로 해석해야 한다.
- `docs/ci.md`는 현재 상태를 "test harness only"로 설명하고 있으므로, Story 1.0 완료 시점에는 앱 셸/계약/perf gate가 반영된 상태로 업데이트해야 한다.
- `.nvmrc`는 `24.14.1`이고 Prisma 최신 시스템 요구사항은 Node `^20.19.0`, `^22.12.0`, `^24.0.0`을 지원하므로 현재 버전대는 유지 가능하다. 다만 `package.json`의 engine 표현과 맞춰야 한다.

### Latest Technical Notes

- Next.js 공식 설치 문서는 App Router 기반 신규 앱 생성 시 TypeScript, `src/` directory, alias 구성을 기본 경로로 지원한다. 이 스토리에서는 해당 구성을 현재 저장소 root에 수동 반영해야 한다.
- Next.js 공식 문서는 `tsconfig.json`의 `baseUrl`/`paths`로 alias를 지원한다. Story 1.0의 `@/*` alias는 여기와 일치해야 한다.
- Prisma 공식 시스템 요구사항 기준 최신 Prisma ORM은 Node `^20.19.0`, `^22.12.0`, `^24.0.0` 이상을 지원한다. 현재 `.nvmrc`의 `24.14.1`은 범위 안이다.
- GitHub protected branches 문서 기준 required status checks는 기본적으로 strict 모드가 기본 동작이므로, Story 1.0에서는 체크 이름을 안정적으로 고정해야 한다.

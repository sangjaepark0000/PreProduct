# Story 1.4: 최소 검증 관측

Status: review

## Story

As a 개발팀,  
I want `프리리스팅` 사용 여부를 확인할 최소 관측만 남기고 싶다,  
so that 과한 계측 시스템 없이도 현재 MVP 가설을 검증할 수 있다.

## Acceptance Criteria

1. 매물 생성 시 `listing.created.v1` 또는 동등한 생성 기록이 남는다.
2. 생성 기록에는 `initialStatus`가 포함된다.
3. `updatedAt` 기준으로 7일 내 수정 여부를 확인할 수 있다.
4. 운영자는 별도 전용 화면 없이도 전체 등록 수, `initialStatus` 분포, 7일 내 수정 수를 단순 조회 또는 export로 확인할 수 있다.

## Tasks / Subtasks

- [ ] 생성 기록과 관측 요약 경계를 추가한다. (AC: 1, 2, 3, 4)
  - [ ] 기존 `Listing` 저장 데이터에서 `listing.created.v1`에 해당하는 최소 생성 기록을 재구성할 수 있는 도메인 경계를 만든다.
  - [ ] 전체 등록 수, `initialStatus` 분포, 7일 내 수정 수를 계산하는 관측 서비스/저장소 경계를 추가한다.
  - [ ] 7일 내 수정 여부는 `updatedAt > createdAt` 이면서 `updatedAt <= createdAt + 7일` 기준으로 고정한다.

- [ ] 운영용 조회/export 엔드포인트를 추가한다. (AC: 4)
  - [ ] 전용 UI 대신 JSON 조회 경로를 제공한다.
  - [ ] 필요 시 CSV export도 같은 경계에서 제공한다.
  - [ ] 응답에는 summary와 생성 기록 export가 함께 포함되거나 일관되게 분리되어야 한다.

- [ ] Story 1.1~1.3 흐름을 깨지 않도록 테스트를 확장한다. (AC: 1, 2, 3, 4)
  - [ ] 관측 서비스 테스트에 total/status distribution/updated-within-7-days 계산을 추가한다.
  - [ ] route 테스트에 JSON/CSV export와 record shape을 고정한다.
  - [ ] 기존 `listing.created.v1` contract 테스트를 유지한다.
  - [ ] `lint`, `typecheck`, `unit`, `contract`, `perf-budget`를 유지한다.

## Dev Notes

### Story Intent

- Story 1.4는 대시보드 같은 별도 화면이 아니라, 운영자가 단순 조회/내보내기로 MVP 가설을 판단할 수 있는 최소 관측만 제공하는 스토리다.
- 이미 `Listing`에 `createdAt`, `updatedAt`, `initialStatus`가 저장되므로, 추가적인 대형 계측 시스템 없이도 운영용 export를 만들 수 있다.

### Recommended Implementation Shape

- 권장 수정 범위:
  - `src/domain/observability/`
  - `src/infra/listing/`
  - `src/app/api/ops/`
  - `tests/contracts/`
  - route/unit test 파일
- 경계 규칙:
  - 생성 기록 재구성은 domain/service에서 수행한다.
  - Prisma 직접 호출은 route가 아니라 infra repository에서만 한다.
  - 운영 조회는 UI 대신 route handler 중심으로 해결한다.

### Testing Requirements

- 서비스 테스트:
  - `initialStatus` 분포 계산
  - 7일 내 수정 수 계산
  - 생성 기록 shape 검증
- route 테스트:
  - JSON summary 응답
  - CSV export 응답
  - 빈 데이터셋에서도 안정 동작

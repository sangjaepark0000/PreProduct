# Story 1.3: 자산 단위 선의향 등록 및 제약조건 편집

Status: done

## Story

As a 판매/구매 고민 사용자,  
I want 자산 단위 의향과 제약조건을 등록·수정할 수 있고 선의향을 공개하며,  
so that 타이밍과 목적에 맞는 초기 판단 데이터를 쌓을 수 있다.

## Acceptance Criteria

1. **Given** 사용자가 자산을 선택한 상태에서  
   **When** 의향, 처분유형, 시간압박, 유동성 필요, 예산/기간 제약을 입력해 저장하면  
   **Then** 의향 레코드가 생성 또는 수정되어야 한다.
2. **Given** 동일 요청이 재전송되었을 때  
   **When** 동일 `x-idempotency-key`로 저장 API를 호출하면  
   **Then** 중복 저장 없이 동일 결과를 반환해야 한다.
3. 의향 저장 계약은 FR3/FR4/FR13/FR14/FR17/FR18 필드를 모두 포함하며 Story 2.x 판단 카드 입력에 재사용 가능한 DTO 구조여야 한다.  
   최소 필드: `intentId`, `assetId`, `intentType`, `disposalIntentType`, `timePressure`, `liquidityNeed`, `budgetConstraint`, `periodConstraint`, `isPublicIntent`, `version`, `updatedAt`, `changeReasonSummary`
4. 저장 API는 `{ data, meta }` / `{ error: { code, message, details?, requestId } }` 규약과 ISO-8601 UTC 시각 규칙을 유지해야 한다.
5. 의향 공개(`FR13/FR14`) 상태와 제약조건 변경 이력이 추적 가능하도록 최소 변경 메타(수정시각, 변경근거 요약)를 포함해야 한다.
6. 이벤트 로깅은 PRD canonical 이벤트(`start_intent_input`, `complete_intent_input`) 기준으로 기록되어야 하며 dedupe 기준을 만족해야 한다.
7. Story 1.2 진단 저장 API(`/api/intents`) 계약/동작은 회귀 없이 유지되어야 한다. 의향 저장 구현은 별도 경계(`/api/intents/records`)로 분리해야 한다.

Coverage: FR3, FR4, FR13, FR14, FR17, FR18

# Story 1.3: 필수 필드 검증 및 재시도 복구

Status: done

## Story

As a 판매 사용자,  
I want 저장 실패나 검증 실패 시 즉시 복구하고 재시도하고 싶다,  
so that 등록 플로우가 중단되지 않는다.

**Scope Tag:** Active MVP  
**Story Type:** Feature  
**FR Trace:** FR16, FR17

## Acceptance Criteria

1. **Given** 필수 필드가 누락되었거나 저장 중 오류가 발생했을 때  
   **When** 사용자가 저장을 요청하면  
   **Then** 누락/오류 원인이 사용자에게 명확히 표시된다  
   **And** 동일 화면에서 수정 후 재시도할 수 있다.

Coverage: FR16, FR17

## Dev Notes

### Epic Context

- Story 1.3은 Story 1.1(생성/임시저장)과 Story 1.2(조회/수정)에서 이미 구현된 listing draft 경계 위에, 실패 복구의 일관성과 사용자 안내 품질을 강화하는 스토리다.
- 이번 스토리는 신규 도메인 추가보다 기존 실패 분기의 정합성과 회귀 안정성을 우선한다.

### Previous Story Intelligence

- Story 1.1에서 필수 필드 검증과 `same_screen_retry` 규약이 도입되었다. Story 1.3은 이를 필드별 오류 가시화까지 확장해야 한다.
- Story 1.2에서 `create/update/reload` 재시도 분기가 분리되었고 stale 상태 정리 패치가 반영되었다. Story 1.3은 이 분기 일관성을 깨지 않도록 동일 구조를 유지해야 한다.
- Story 1.2 리뷰에서 확인된 핵심 리스크:
  - 재조회 실패 후 stale draft 잔존
  - 액션 타입과 다른 재시도 실행
  - keySpecs 직렬화/역직렬화 경계 손상
  위 항목은 Story 1.3에서 회귀 금지 포인트다.

### Current Codebase Intelligence

- listing draft 구현은 이미 `contracts -> mapper -> domain -> route -> UI -> e2e` 구조를 따른다.
- 생성/수정/재조회 API는 공통적으로 `{ data, meta }` / `{ error }` 계약을 사용한다.
- `x-preproduct-owner-id` 기반 서버 컨텍스트 정책(비프로덕션 fallback 포함)이 존재하므로 Story 1.3에서 권한 모델은 변경하지 않는다.

### Architecture Compliance (Must Follow)

- 레이어 경계: `feature -> domain -> infra`만 허용, `feature -> infra` 직접 import 금지
- 매핑 경계: casing 변환은 `src/infra/mapper` 전용
- API 응답 형식: 성공 `{ data, meta }`, 실패 `{ error: { code, message, details?, requestId } }`
- Scope 우선순위: Active MVP > Deferred P1.5+ > Legacy Reference
- 변경 순서: `contracts -> mapper/domain -> routes -> UI -> tests`

### Technical Requirements

- 필수 필드 검증 대상: `title`, `category`, `keySpecs(1+)`, `price(>0)`
- 오류 상세는 "무엇이 잘못되었는지"와 "어떻게 복구하는지"를 함께 제공해야 한다.
- 동일 화면 재시도는 입력 유지가 기본이며, 성공 전까지 사용자 입력이 손실되면 안 된다.
- `updatedAt`/`createdAt` 불변 규칙 및 소유권(403) 규칙은 기존 동작을 유지해야 한다.
- NFR 연계: 핵심 액션 p95 2초, WCAG AA, 키보드 접근/상태 알림 유지

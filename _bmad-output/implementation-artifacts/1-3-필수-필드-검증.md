# Story 1.3: 필수 필드 검증

Status: ready-for-dev

## Story

As a 판매 사용자,  
I want 필수 입력 오류를 이해하고 즉시 고치고 싶다,  
so that 등록 플로우가 중단되지 않는다.

## Acceptance Criteria

1. 필수 필드 누락 시 오류 원인이 명확히 표시된다.
2. 수정 후 동일 화면에서 다시 저장할 수 있다.
3. 모바일에서도 오류와 다음 행동이 명확히 보인다.

## Dev Notes

### Story Intent

- Story 1.3은 Story 1.1/1.2에서 이미 동작하는 등록/상태 저장 흐름을 깨지 않으면서, 필수 입력 실패를 더 빠르게 이해하고 복구하게 만드는 데 집중한다.
- 저장 실패 후 같은 화면에서 다시 시도하는 패턴은 유지하되, 오류의 원인과 다음 행동을 더 명확히 드러내야 한다.

### Current Workspace Reality

- 현재 `/listings/new`는 서버 액션 기반으로 같은 화면 복구가 가능하지만, 상단 요약이 없어 모바일에서 어떤 필드를 먼저 고쳐야 하는지 한 번에 파악하기 어렵다.
- `src/domain/listing/listing.ts`의 현재 메시지는 최소 수준이며, 가격 입력은 "비어 있음"과 "0/음수/비숫자"를 더 구분할 여지가 있다.
- Story 1.2의 상태 선택/상세 수정 플로우는 이미 `origin/main`에 병합돼 있으므로, Story 1.3은 생성 화면 검증 품질을 우선 강화한다.

### Recommended Implementation Shape

- 권장 수정 범위:
  - `src/domain/listing/`
  - `src/feature/listing/actions/create-listing.action.ts`
  - `src/feature/listing/components/listing-form.client.tsx`
  - `src/feature/listing/components/listing-submit-bar.client.tsx`
  - `tests/e2e/listing-registration.spec.ts`
- 경계 규칙:
  - `feature -> domain -> infra`만 허용
  - 페이지/컴포넌트에서 직접 검증 규칙을 중복 정의하지 않는다.
  - 저장 실패 재시도 규약은 Story 1.1 패턴을 유지한다.

### UX and Accessibility Guardrails

- 오류는 상단 요약 + 필드별 helper text로 함께 보여야 한다.
- 요약 메시지는 "무엇이 잘못됐는지"와 "다음에 뭘 해야 하는지"를 같이 말해야 한다.
- 모바일에서도 첫 화면 안에 오류 상태가 들어오도록 과도한 장문 설명은 피한다.
- 포커스/라벨/오류 helper 연결은 MUI 기본 접근성 패턴을 유지한다.

### Testing Requirements

- 액션 테스트:
  - 제목/카테고리/핵심 스펙/가격 누락 메시지 검증
  - 가격 형식 오류 검증
  - 저장 실패 후 값 유지 검증
- 브라우저 테스트:
  - 빈/불완전 입력 제출 시 같은 화면에서 오류 요약 확인
  - 수정 후 재제출 성공
